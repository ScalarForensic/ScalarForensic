## 3. Finding 1 — CUDA Unavailable (PyTorch/Driver Version Mismatch)

### Observation

On first run, PyTorch reported CUDA unavailable:

```
UserWarning: CUDA initialization: The NVIDIA driver on your system is too old
(found version 12080). Please update your GPU driver.
backend=SSCDEmbedder  dim=512  device=cpu
```

Embedding throughput: **~6.7 img/s** (DINOv2 on CPU).

### Root cause

The warning message was misleading. The driver (570.211.01, CUDA 12.8) was current. The real mismatch was the opposite direction: uv had resolved `torch==2.11.0+cu130`, a build compiled for CUDA 13.0, which requires a newer driver than installed.

```
Installed:  torch 2.11.0+cu130  (requires CUDA 13.0+ driver)
Driver cap: CUDA 12.8
Result:     CUDA init fails, falls back to CPU
```

### Secondary issue — cuDNN symbol error

After switching to the correct PyTorch build, a second error appeared:

```
Invalid handle. Cannot load symbol cudnnGetVersion
```

The cuDNN libraries shipped inside the venv (`nvidia-cudnn-cu12`) were not on `LD_LIBRARY_PATH`, so PyTorch could not locate them at runtime.

### Fix

**1. Pin PyTorch to the cu128 wheel index** in `pyproject.toml`:

```toml
[[tool.uv.index]]
name = "pytorch-cu128"
url = "https://download.pytorch.org/whl/cu128"
explicit = true

[tool.uv.sources]
torch = [{ index = "pytorch-cu128" }]
torchvision = [{ index = "pytorch-cu128" }]
```

**2. Set `LD_LIBRARY_PATH`** via a project-local wrapper `run.sh` (no system-level changes):

```bash
#!/usr/bin/env bash
NVIDIA_BASE="$SCRIPT_DIR/.venv/lib/python3.14/site-packages/nvidia"
_cuda_paths=$(find "$NVIDIA_BASE" -maxdepth 2 -name "lib" -type d | tr '\n' ':')
export LD_LIBRARY_PATH="${_cuda_paths}${LD_LIBRARY_PATH}"
exec uv run "$@"
```

### Verification

```
torch: 2.11.0+cu128
CUDA available: True
GPU: NVIDIA GeForce RTX 4060 Ti
```

All subsequent runs use `./run.sh` in place of `uv run`.

## 4. Finding 2 — Redundant Image Preprocessing (DINOv2 Double Resize)

### Observation

`DINOv2Embedder.normalize_batch_bytes` resized every image to 512×512 using PIL/LANCZOS before passing it to HuggingFace `AutoImageProcessor`. Inspection of the processor configuration revealed:

```
size: shortest_edge=256
crop_size: 224×224
do_resize: True
do_center_crop: True
```

The processor always resizes the shortest edge to 256 px and center-crops to 224×224, regardless of input size. The 512×512 PIL resize was therefore discarded immediately, wasting CPU time and introducing a second resampling artefact.

### Model input requirements

| Model | Pipeline | Final input |
|-------|----------|-------------|
| SSCD | short side → 331 px → center crop | 288×288 |
| DINOv2 | short side → 256 px → center crop (via AutoImageProcessor) | 224×224 |

The binding constraint is SSCD at **331 px short side**. Any image pre-capped to 331 px satisfies both models — DINOv2's processor can downscale from 331 to 256 trivially.

### Fix

Introduced a shared `preprocess_batch()` function executed once per batch before either model:

1. Open PIL image from bytes (parallelised with `ThreadPoolExecutor`)
2. If short side > 331 px: scale down to 331 px preserving aspect ratio (LANCZOS)
3. If short side ≤ 331 px: leave untouched (model will upscale as required)

Per-model `normalize_batch_bytes` updated:

- **SSCD:** receives pre-capped PIL image → performs only the center crop to 288×288 (resize is now a near-no-op for large images)
- **DINOv2:** receives pre-capped PIL image → returns it unchanged; `AutoImageProcessor` handles all resizing

This eliminates the redundant resize, halves the number of PIL `open()` calls, and enables the shared pre-step to be parallelised independently.

## 5. Finding 3 — Sequential Pipeline Bottleneck

### Observation

The original per-batch execution order was strictly sequential:

```
read → hash → [SSCD normalize → SSCD embed → SSCD upsert] → [DINOv2 normalize → DINOv2 embed → DINOv2 upsert]
```

SSCD normalization ran while DINOv2 waited. Both Qdrant upserts (independent network writes) ran one after the other.

### Fix

Restructured into three stages:

1. **Shared pre-processing** (threaded, CPU): `preprocess_batch()` over all unique images using `ThreadPoolExecutor` — PIL decode and cap happen in parallel across CPU cores.
2. **Per-model embed loop** (sequential, GPU): normalize (trivial after pre-cap) → embed. GPU is shared so embed steps remain sequential.
3. **Parallel upserts** (threaded, network): both Qdrant upsert calls dispatched concurrently via `ThreadPoolExecutor`.

Resulting batch line format:

```
batch N [X imgs  Y MB  Z img/s total]  read Xs  hash Xs  pre Xs
  |  SSCDEmbedder norm Xs  embed Xs (R img/s)  +N
  |  DINOv2Embedder norm Xs  embed Xs (R img/s)  +N
  |  upsert Xs
```

## 6. Finding 4 — GPU Underutilisation (fp32 Inference)

### Observation

Benchmark across batch sizes 16–256 showed essentially flat throughput:

| Batch | img/s | DINO embed |
|-------|-------|------------|
| 16 | 48.1 | 0.272 s |
| 32 | 48.4 | 0.555 s |
| 64 | 48.8 | 1.096 s |
| 128 | 49.0 | 2.189 s |
| 256 | 48.5 | 4.435 s |

DINOv2 embed time scaled linearly with batch size (~57.7 img/s constant), confirming the GPU was compute-saturated at all batch sizes. `nvidia-smi` confirmed 100% GPU utilisation at 136 W / 165 W TDP.

**Root cause:** DINOv2-large was loaded in fp32 (~4.4 GB VRAM). The RTX 4060 Ti Tensor Cores deliver significantly higher throughput for fp16.

### Fix

Two changes applied to both embedders when running on CUDA:

**1. fp16 inference:** Load model weights directly in `torch.float16`; cast input tensors to `.half()` before the forward pass; convert output embeddings back to fp32 (`.float()`) before returning/upserting.

```python
# DINOv2
dtype = torch.float16 if self.device == "cuda" else torch.float32
self.model = AutoModel.from_pretrained(model_name, dtype=dtype).to(self.device)

# SSCD
self._model = torch.jit.load(model_name, map_location=self.device)
if self.device == "cuda":
    self._model = self._model.half()
```

**2. `torch.compile()`:** Enables kernel fusion and op scheduling optimisation. Applied after model load on CUDA. First batch incurs a one-time JIT compilation delay (~10–30 s); all subsequent batches benefit.

```python
if self.device == "cuda":
    self.model = torch.compile(self.model)
```

### Benchmark results after fix

| Batch | img/s | DINO embed | SSCD embed |
|-------|-------|------------|------------|
| 16 | 135.3 | 0.085 s | 0.027 s |
| 32 | 133.7 | 0.173 s | 0.055 s |
| 64 | 136.3 | 0.337 s | 0.115 s |
| **128** | **136.4** | **0.671 s** | **0.237 s** |
| 256 | 135.4 | 1.350 s | 0.481 s |

**Result: 48.5 → 136.4 img/s — 2.35× improvement.**  
VRAM usage: ~4.4 GB (fp32) → ~2.2 GB (fp16), freeing ~2.2 GB headroom.

Optimal batch size set to **128** (`SFN_BATCH_SIZE=128` in `.env`).

## 7. Inference Dtype Consistency Note

Stored vectors in Qdrant are always fp32 (output embeddings are cast with `.float()` before upserting). The fp16 inference affects only intermediate computation, not stored values.

However, the same image processed in fp16 vs fp32 will produce embedding vectors that differ by approximately `1e-3`. For cosine similarity ranking this difference is negligible and does not affect retrieval quality in practice.

**Implication for audit trails:** Any collection indexed under fp32 inference and re-queried under fp16 inference will produce consistent similarity rankings but not bit-exact vector reproduction. For maximum reproducibility, all indexing and querying should use the same inference dtype. Since the existing collections were partially indexed under fp32 (before this session's fixes), a full re-index under fp16 is recommended and planned.

The inference dtype is not currently stored as payload metadata per Qdrant point. Adding `inference_dtype` to the point payload is recommended for future audit traceability.

## 8. Benchmark Script

A standalone benchmark script is available at `analysis_test/benchmark_batch_size.py`. It loads both models once, runs the full embedding pipeline (pre-process → normalize → embed) at each requested batch size with configurable repetitions, and reports a throughput table. Upsert is excluded to isolate GPU embedding performance.

**Usage:**

```bash
./run.sh python analysis_test/benchmark_batch_size.py
./run.sh python analysis_test/benchmark_batch_size.py --sizes 64,128,192,256 --repeats 5
```

## Summary of Changes

| Area | Before | After |
|------|--------|-------|
| PyTorch build | `cu130` (wrong) | `cu128` (matches driver) |
| CUDA libs | Not on `LD_LIBRARY_PATH` | Exposed via `run.sh` wrapper |
| DINOv2 preprocessing | PIL resize 512×512 + AutoImageProcessor | Shared 331px cap + AutoImageProcessor only |
| SSCD preprocessing | Full resize from raw bytes | Shared 331px cap + center crop only |
| PIL decode | Sequential, per-model | Threaded `ThreadPoolExecutor`, shared |
| Upsert | Sequential (SSCD then DINOv2) | Parallel `ThreadPoolExecutor` |
| Inference precision | fp32 | fp16 (CUDA only) |
| Kernel optimisation | None | `torch.compile()` (CUDA only) |
| Batch size | 32 | 128 |
| Throughput | ~6.7 img/s (CPU) / ~48 img/s (fp32 GPU) | **136 img/s (fp16 + compiled GPU)** |
